# モデレーション運用手順

通報機能（PR #35 / 設計 #34）に基づく運用手順。通報データの取扱い、自動措置、人手レビュー、異議申立を定める。

- 最終更新日: 2026-07-16
- 関連 issue: #38（問い合わせ窓口）／ #34（設計）／ #24（将来の集計 D1 化）

## 1. 通報の仕組み（Phase 1）

- 利用者は通話画面の旗アイコンから、参加者を理由付きで通報できる（理由: ハラスメント／ヘイト／スパム／不適切な名前／違法・チート／その他）。
- 通報と同時に、**通報者の端末で対象をローカルミュート**する（通常のミュートと同じ扱い。後から解除可）。
- 通報は `POST /api/sessions/:id/reports` で記録する。送信失敗は通話をブロックしない。

## 2. 保存とプライバシー

- 通報者・被通報者の Summoner ID は **HMAC でハッシュ化**して保存（生 ID は保存しない）。
- キー: `report:{reportedHash}:{sessionId}:{reporterHash}`（1 セッション 1 ペア 1 件で連投を無効化）。TTL 30 日。
- metadata に `reporter / reason / time` を保持し、集計を軽量化する。

## 3. スコアリングと自動 BAN（Phase 2/3）

- 集計はサーバ側で自動実行（`functions/api/_lib/moderation.ts`）。`waitUntil` によりレスポンスを遅延させない。
- **リスクスコア = ユニーク通報者ごとの（重大度 × 時間減衰）の合計**。
  - 重大度: ハラスメント/ヘイト/違法 = 3、スパム = 2、不適切な名前/その他 = 1。
  - 時間減衰: 7 日未満 = 1.0、30 日未満 = 0.5、それ以降 = 0（＝失効。ユニーク通報者数にも数えない）。
- **自動 BAN の条件**: ユニーク通報者 **3 人以上** かつ スコア **7 以上**。
  - 単独の通報者による連投では BAN されない（虚偽通報耐性）。
- BAN は `ban:{reportedHash}`、**24 時間の一時措置**（ソフト）。join 時に BAN 済み identity を 403 で拒否する。

### しきい値の調整

`functions/api/_lib/moderation.ts` の定数（`MIN_DISTINCT_REPORTERS` / `BAN_SCORE_THRESHOLD` / `AUTO_BAN_SECONDS` / `SEVERITY`）で調整する。運用開始後、誤検知率を見て見直す。

## 4. 人手レビュー・フロー

1. 通報の傾向（同一 `reportedHash` への集中、重大度）を定期的に確認する。
2. 自動 BAN は一時措置に留め、恒久 BAN は行わない（誤検知・組織的虚偽通報への配慮）。
3. 悪質性が明確な場合のみ、手動で BAN（`ban:` キーを付与、`source: "manual"`）を検討する。

## 5. 異議申立

- 一時 BAN を受けた利用者は、**問い合わせ窓口**（設置は #38）から異議を申し立てられる。
- 受付 → 事実確認（通報傾向・状況） → 必要に応じ BAN 解除（`ban:` キー削除）または期間短縮。

## 6. 限界と今後

- 現状 identity は自己申告の Summoner ID のため、BAN は名前変更で回避可能。**RSO（#19）で本人確認が入って初めて実効性が高まる**。それまではソフトな抑止に留める。
- 通報の本格集計は KV から D1 へ移行予定（#24）。
