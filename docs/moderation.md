# モデレーション運用手順

通報機能（PR #35 / 設計 #34）に基づく運用手順。通報データの取扱い、自動措置、人手レビュー、異議申立を定める。

- 最終更新日: 2026-07-19
- 関連 issue: #38（問い合わせ窓口・法務サインオフ）／ #34（設計）／ #24（集計の D1 化 → 実施済み）

## 1. 通報の仕組み（Phase 1）

- 利用者は通話画面の旗アイコンから、参加者を理由付きで通報できる（理由: ハラスメント／ヘイト／スパム／不適切な名前／違法・チート／その他）。
- 通報と同時に、**通報者の端末で対象をローカルミュート**する（通常のミュートと同じ扱い。後から解除可）。
- 通報は `POST /api/sessions/:id/reports` で記録する。送信失敗は通話をブロックしない。

## 1.5 通報を受け付ける条件（濫用防止）

`POST /api/sessions/:id/reports` は、次をすべて満たす場合のみ受理する。

1. **ルームが実在する**（`game:{id}` → `session:{meetingId}` の両方が KV にある）。無い場合 404。
2. **通報者・被通報者の双方が、そのルームの名簿（`session.users`）に載っている**。載っていなければ 403。突合は正規化（trim + 小文字化）した Riot ID で行う。
   - 名簿は追記のみで、退出しても消えない。したがって**退出済みの相手も通報できるし、自分が退出した後でも通報できる**。これは意図した挙動。
3. **同一 IP の通報が直近 10 分で 5 件未満**（`functions/api/_lib/rate-limit.ts`）。超過は 429。

**なぜ必要か**: 2 の検証を入れるまで、`reporterSummonerId` はリクエストボディ中の未検証の文字列だった。つまり攻撃者が 1 人で、架空の通報者名 3 つを使って `harassment`（重み 3）を 3 件 POST すれば、ユニーク通報者 3 人・スコア 9 が成立し、**任意の Riot ID を 24 時間 BAN できた**。名簿検証によって、通報者を 3 人揃えるには「実在の 3 アカウントが実際に同じルームに参加している」ことが必要になる。

**残る限界**: 自己申告 ID である以上（RSO #19 まで）、同じルームに入った 3 人が結託すれば 4 人目を BAN できることは防げない。レート制限は KV のアトミック更新が無いため read-modify-write で、同時リクエストでは数件取りこぼす（第二の防御線という位置づけ。第一は名簿検証）。

### 自動 BAN のキルスイッチ

環境変数 `MODERATION_AUTO_BAN_ENABLED="false"` で、**自動 BAN だけを止める**（通報の記録・ローカルミュートは従来どおり動く）。武器化のパターンが修正の出荷より速く現れた場合の緊急停止用。設定方法は [deploy-and-secrets.md](./deploy-and-secrets.md)。

## 2. 保存とプライバシー

- 通報は **D1（`voicecrew-reports` / バインディング `VC_DB`）の `reports` テーブル**に保存する。スキーマは `migrations/0001_create_reports.sql`。
- 通報者・被通報者の Riot ID は **平文で保存**し、**TTL を設けず恒久保存**する（2026-07-17 の決定）。
  - 突合用に正規化（trim + 小文字化）した値と、表示用の生文字列の両方を持つ。
  - 一意制約 `(reported_riot_id, session_id, reporter_riot_id)` により 1 セッション 1 ペア 1 件。再通報は UPSERT で上書きし、連投でスコアを膨らませられない。
- ハッシュ化は廃止した（`REPORT_SALT` / `hashId()` も削除済み）。理由は異議申立時に根拠記録を照合して回答すること、CSAE の証跡保全、開示請求への応答。
  - 自己申告 ID のため「本人確認」は成立しない（RSO #19 まで）。ポリシーも「本人確認」ではなく「根拠記録の照合・回答」と書いてある。混同しないこと。
- **この取扱いはプライバシーポリシー 1/5/7/8/9/10/11 条と利用規約 3/6 条に明記されている。実装を変えるときはポリシー文言も併せて変えること**（片方だけ変えると虚偽表示になる）。
- BAN のみ KV に残る（`ban:{正規化 Riot ID}`、24 時間で自動失効）。一時措置であり恒久保存の対象ではない。

## 3. スコアリングと自動 BAN（Phase 2/3）

- 集計はサーバ側で自動実行（`functions/api/_lib/moderation.ts`）。`waitUntil` によりレスポンスを遅延させない。
- **リスクスコア = ユニーク通報者ごとの（重大度 × 時間減衰）の合計**。
  - 重大度: ハラスメント/ヘイト/違法 = 3、スパム = 2、チート/不適切な名前/その他 = 1。
  - 時間減衰: 7 日未満 = 1.0、30 日未満 = 0.5、それ以降 = 0（＝失効。ユニーク通報者数にも数えない）。
  - **通報は恒久保存になったため、30 日超の通報を無効化しているのはこの時間減衰だけ**である。減衰を外すと、古い通報が永久に累積して BAN が積み上がる。減衰の変更は BAN の意味を変えるので慎重に。
- **自動 BAN の条件**: ユニーク通報者 **3 人以上** かつ スコア **7 以上**。
  - 単独の通報者による連投では BAN されない（虚偽通報耐性）。ただしこの耐性が実際に成立するのは、**通報者がルーム名簿で検証されている前提の下だけ**である（1.5 を参照）。検証を外すとこの条件は無意味になる。
- BAN は `ban:{正規化 Riot ID}`、**24 時間の一時措置**（ソフト）。join 時に BAN 済み identity を 403 で拒否する。

### しきい値の調整

`functions/api/_lib/moderation.ts` の定数（`MIN_DISTINCT_REPORTERS` / `BAN_SCORE_THRESHOLD` / `AUTO_BAN_SECONDS` / `SEVERITY`）で調整する。運用開始後、誤検知率を見て見直す。

## 3.5 子どもの安全（CSAE）に関する通報

通報理由「子どもの安全（性的な接触・搾取）」は、他の理由と**別扱い**とする。

- 保存先は他の通報と同じ `reports` テーブル（`reason = 'child_safety'`）。全通報が恒久保存になったため、従来の 1 年 TTL 付き `csae:` キーは廃止した（保全期間はむしろ延びる）。
- **自動スコア/自動 BAN には投入しない**。単独の通報者でもエスカレーションできる必要があり、かつ最重大カテゴリを score ベースの自動措置に載せると虚偽通報の武器化を招くため。除外は `evaluateReports()` 側で行う（`SEVERITY` に `child_safety` を載せないこと。載せるとフォールバックの重み 1 で加算されてしまう）。
- 受領した CSAE 通報は**人手で最優先レビュー**する。明白な事案では手動措置（接続拒否等）を行う。
- 各法域で、事業者に通報・保全義務が生じ得る（例: 米 18 U.S.C. §2258A/NCMEC、EU DSA、日 児ポ法・プロバイダ責任制限法）。**具体的な義務範囲と当局窓口は有資格弁護士に確認する**（音声を録音しないため、保全対象は通報メタデータ・当事者の Riot ID・時刻等）。

## 3.6 平文・恒久保存に伴う管理

ハッシュ化を廃止したため、`reports` テーブルは **それ自体が個人情報の平文データベース**である。漏えい＝当事者の Riot ID の直接露出であり、ハッシュによる緩和はもう存在しない。

- D1 へのアクセス権限は最小限に絞る。エクスポート（`wrangler d1 export`）の取扱いに注意する。
- 恒久保存のため、**削除は運用でしか起きない**。開示・削除請求の受付フロー（#38）が実質的な唯一の消去手段になる。
- **未解決（有資格弁護士のサインオフが #38 で未了）**: 要配慮個人情報の取得同意、GDPR 10 条（犯罪嫌疑データ）の授権、Riot への第三者提供の同意、韓国 PIPA の破棄義務と無期限保存の衝突。いずれも**ポリシーに書けば適法になるという性質のものではない**。

## 4. 人手レビュー・フロー

1. 通報の傾向（同一 `reported_riot_id` への集中、重大度）を定期的に確認する。D1 に直接クエリできる:
   ```bash
   npx wrangler d1 execute voicecrew-reports --remote --command \
     "SELECT reported_riot_id_raw, reason, COUNT(*) c FROM reports GROUP BY 1,2 ORDER BY c DESC LIMIT 20"
   ```
2. 自動 BAN は一時措置に留め、恒久 BAN は行わない（誤検知・組織的虚偽通報への配慮）。
3. 悪質性が明確な場合のみ、手動で BAN（`ban:` キーを付与、`source: "manual"`）を検討する。

## 5. 異議申立

- 一時 BAN を受けた利用者は、**問い合わせ窓口**（設置は #38）から異議を申し立てられる。
- 受付 → 事実確認（通報傾向・状況） → 必要に応じ BAN 解除（`ban:` キー削除）または期間短縮。

## 6. 限界と今後

- 現状 identity は自己申告の Summoner ID のため、BAN は名前変更で回避可能。**RSO（#19）で本人確認が入って初めて実効性が高まる**。それまではソフトな抑止に留める。
  - 同じ理由で、**平文保存された Riot ID が本人のものである保証もない**。第三者の Riot ID を騙って通報された記録が、その第三者の名前で恒久的に残り得る。異議申立時の本人確認という保存目的も、RSO が入るまでは十分に達成されない。
- 通報集計の D1 化（#24）は完了。
- Riot Games への通報連携は**未実装、かつ公開ポリシーには一切書いていない**（運営者の方針で非記載）。むしろ現ポリシー第 5 条は「通報データは Riot ID 検証の送信対象に含まない」と積極的に述べている。
  - したがって将来連携を始める場合、単なる追記では済まず、利用目的の変更・第三者提供の同意・越境提供の要件が発生し、**既存の通報データは遡って連携できない**（取得時の目的外利用になる）。実装前に必ず法務確認を通すこと。
